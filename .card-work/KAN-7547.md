### 🎯 Visão Geral e Justificativa Técnica

A análise realizada identificou que o arquivo `server.cjs` opera como um 'God Class' (Classe de Deus), misturando inicialização do servidor, lógica de roteamento (`routes`), manipulação de banco de dados (`db calls`) e tratamento de erros. Essa arquitetura monolítica compromete a manutenibilidade, dificulta os testes unitários e introduz riscos de *race conditions* em cenários de alto tráfego.

Esta refatoração visa aplicar princípios de **Separação de Responsabilidades (SoC)** e utilizar recursos assíncronos do Node.js de forma idiomática para garantir escalabilidade, performance previsível e observabilidade robusta.

### ✨ Detalhamento das Melhorias Aplicadas

1.  **Modularização Profunda:** O código foi desacoplado em módulos específicos:
    *   `server.cjs`: Agora só é responsável por *inicializar* a aplicação, carregar configurações e iniciar o servidor HTTP.
    *   `routes/apiRoutes.js`: Contém apenas os endpoints de roteamento (ex: `router.get('/users', ...)`), sem saber como o banco de dados opera.
    *   `services/userService.js`: Contém a lógica de negócio pura (Core Logic). É aqui que as transações complexas devem ocorrer, isoladas do Express e do acesso direto ao DB.
    *   `utils/errorMiddleware.js`: Middleware centralizado para tratamento de erros em nível global, garantindo respostas HTTP padronizadas (e sem vazar stack traces internos).
2.  **Gerenciamento Assíncrono:** Substituição de quaisquer operações potencialmente síncronas ou baseadas em *callbacks* aninhados por padrões `async/await` com `try...catch` robustos. Isso aumenta a legibilidade e o desempenho, melhorando drasticamente a capacidade de concorrência.
3.  **Configuração Centralizada:** Implementação do uso de variáveis de ambiente (`dotenv` ou similar) para todas as configurações sensíveis (portas, credenciais DB). Nenhuma constante mágica foi hardcoded.
4.  **Inicialização Estruturada:** Adotou-se um padrão `initializeApp()` que garante a ordem correta de inicialização: Config -> Conexão com DB -> Roteamento -> Start do Servidor. Isto facilita o *teardown* (desconexão) em caso de encerramento controlado da aplicação.
