## Relatório Técnico: KAN-6593 - Jira/Git Workflow Automation Service

**Status:** Concluído e Refatorado.
**Arquitetura Alvo:** Implementação de um *Service Layer* robusto, seguindo o padrão Hexagonal (Ports and Adapters), garantindo que a lógica de negócio não dependa diretamente da implementação dos clientes externos (Jira/Git).

### Objetivos Alcançados:
1. **Robustez e Resiliência:** Implementação de mecanismos avançados de *Retry* com backoff exponencial para chamadas de API externas, mitigando falhas temporárias (rate limiting ou indisponibilidade transitória do endpoint).
2. **Idempotência Enforçada:** O fluxo agora garante que a execução múltipla da mesma ação (ex: um push duplo acionando o mesmo PR) não resulte em estados inconsistentes no Jira. Foi introduzido o conceito de `WorkflowContext` para rastrear o estado processado.
3. **Separação de Responsabilidades:** O módulo de lógica principal (`WorkflowCoordinator`) foi desacoplado do cliente API específico (`JiraClient`). Isso facilita a troca de backends (ex: migração de Jira Cloud para Data Center) sem alterar a lógica de negócio central.

### Principais Alterações Arquiteturais:
* **`@TransactionScope`:** Adicionado um decorator/context manager que encapsula as operações críticas. Se qualquer etapa falhar (conexão, validação de permissão, atualização do Jira), o bloco inteiro é revertido ou marcado como pendente com logging detalhado.
* **Enumeração de Estados:** Substituímos a manipulação de strings para status por enums fortes (`WorkflowStatus`) em toda a camada de modelo, prevenindo erros de digitação e melhorando a tipagem. 

**Nota Técnica sobre o Refatoramento:** O uso do *Circuit Breaker* (disjuntor) foi implementado no `JiraClient` para evitar ataques de inundação contra APIs externas caso elas estejam em falha generalizada, permitindo que o sistema opere no modo 'fail-safe' e notifique proativamente os operadores.
