## Análise do Motor Gemma 4
O componente `JiraDashboard` está atualmente realizando múltiplas responsabilidades: 1) Fetching complexos dados (Issues e Commits), 2) Gerenciamento do estado completo de um formulário de criação (`newSummary`, `newDescription`, etc.) e 3) Lógica de submissão. Essa mistura viola o Princípio da Responsabilidade Única (SRP), tornando o componente difícil de testar, manter e escalar (mantientific). Sugere-se refatorar a seção 'Criação de Issue' para um componente filho dedicado (`IssueCreationForm`). Além disso, as props complexas no `JiraIssue` devem ser analisadas em conjunto com a API Gateway para simplificar o contrato de dados que chega ao frontend, reduzindo o acoplamento e melhorando a performance de renderização. Critérios de Aceitação: O componente pai deve apenas orquestrar os estados; a submissão do formulário deve residir isoladamente.
